# Folio+ — Notes pour Claude Code

## Fix tags catalogue + suppression F-11 + renommage "Critique" (13/07) — terminée

Trois correctifs sur le catalogue et les demandes d'accès confidentiel, affinés en plusieurs allers-retours (branche `fix/tags-acces-sensibilite`).

- **Tags catalogue** : `projects_catalog_view` agrégeait les tags avec `array_agg(DISTINCT ...)` sans `ORDER BY` explicite dans l'agrégat — l'ordre de sortie n'était pas garanti déterministe par Postgres, et `ProjectCard.tsx` n'affichait qu'1 tag par catégorie (type/outil/mot-clé). Résultat concret : un tag pouvait ne jamais apparaître sur une carte, quel que soit le rôle du visiteur (pas un bug spécifique à un persona). Fix en deux parties (migration `20260713100000_order_tags_in_catalog_view.sql`) : `array_agg(DISTINCT tr.name ORDER BY tr.name)` (même colonne que le `DISTINCT`, seule syntaxe autorisée) pour `tools`/`keywords`/`types`, et `ProjectCard.tsx` passe de `slice(0, 1)` à `slice(0, 2)` sur ces 3 catégories. **`security_invoker = true` reposé explicitement après le `CREATE OR REPLACE VIEW`** (vérifié en base après coup) — piège déjà rencontré sur cette même vue le 10/07, cf. section "Reste à faire" plus bas.
- **F-11 (auto-approbation par sensitivité) supprimée, pas juste supervisée** — un gap avait été identifié (Léa n'avait aucune trace des demandes auto-validées pour les superviser après coup), mais la piste initiale envisagée (ajouter une action de révocation manuelle) entrait en tension avec la règle produit "l'accès est permanent, pas d'expiration individuelle", et ajoutait un nouveau statut + webhook + UI dédiés à 5 jours du rendu. Décision finale : **retirer entièrement l'auto-approbation**. Toute demande d'accès, quel que soit `sensitivity_level`, passe désormais systématiquement par `status = 'pending'` puis validation manuelle de Léa — plus simple, plus sûr, et le contrôle qu'elle croyait déjà avoir devient réel par construction.
  - Côté client (`AccessRequestModal.tsx`) : retrait du calcul `autoApprove`/`existingApproved` et du spread conditionnel `status: "approved"` à l'insert — chaque ligne `access_requests` repose désormais uniquement sur le défaut de colonne `status = 'pending'`.
  - Côté base (migration `20260713100500_remove_f11_auto_approval.sql`) : revert de la policy `access_requests_insert_pending` (élargie le 10/07 pour autoriser un insert direct en `status = 'approved'` sous conditions F-11) à `status = 'pending'` obligatoire — sans ce revert, un appel direct à l'API (hors UI) aurait pu encore auto-valider même après le retrait du code client. Les deux fonctions `security definer` `has_other_approved_access_request()`/`project_is_sensible()`, qui ne servaient qu'à ce check, ont été supprimées avec la policy (aucune autre référence dans le code).
  - **Section F-11 plus haut dans ce fichier (auto-approbation par projet dans `AccessRequestModal.handleSubmit`) devenue obsolète** — ne plus s'y référer comme comportement actuel, l'historique est laissé tel quel pour la trace.
- **"Très sensible" → "Critique"** : renommage tranché en **libellé d'affichage uniquement** (`src/lib/sensitivityLabels.ts`, `tres_sensible: "Critique"`), même pattern que `formatSecteur()`/`secteurLabels.ts` — décidé à 5 jours du rendu pour éviter tout risque de migration d'enum. L'enum Postgres `sensitivity_level` garde `tres_sensible` en interne partout (schéma, RLS, code) ; `sensitivity_level` reste par ailleurs purement informatif (badge pour Léa), sans aucune logique de validation qui en dépende — cohérent avec la suppression de F-11 ci-dessus.
- **Bug supplémentaire trouvé en testant le fix des tags avec la persona Sophie (pending, accès F-12 approuvé sur des projets confidentiels précis)** : `project_tools_select`/`project_keywords_select`/`project_types_select` (les 3 tables de jonction consommées par `projects_catalog_view`) ne vérifiaient que `get_my_role() = 'validated_visitor'` pour un projet confidentiel — jamais l'accès individuellement approuvé (`access_requests.status = 'approved'` pour ce projet précis), contrairement à `projects_select_unified` sur la table `projects` elle-même. Résultat concret : Sophie voyait bien ses projets confidentiels approuvés dans le catalogue (la ligne `projects` passe), mais aucun de leurs tags (tools/keywords/types), leurs 3 tables de jonction étant filtrées par une RLS qui ignorait ce cas. Migration `20260713101000_fix_project_tags_visibility_for_approved_access.sql` : ajout de la même branche `EXISTS (... access_requests ... status = 'approved')` que `projects_select_unified` aux 3 policies. Pas de risque de récursion croisée (le piège déjà documenté plus bas dans ce fichier) : ces policies interrogent `projects` et `access_requests`, mais ni `access_requests_select` ni `projects_select_unified` ne référencent en retour ces 3 tables de jonction.
- **Deuxième trou dans la même RLS, trouvé juste après en retestant** : `projects_select_unified` autorise aussi la visibilité "teaser" d'un projet confidentiel pour un visiteur `pending`/anonyme (avant toute demande d'accès — c'est ce qui affiche la carte floutée avec titre/description/tags dans `ProjectCard.tsx`), mais les 3 policies de jonction n'avaient toujours pas cette branche après le fix ci-dessus (juste ajouté le cas "accès approuvé"). Résultat : un visiteur `pending` sans aucune demande sur un projet confidentiel ne voyait aucun tag sur sa carte teaser, alors que titre/description/secteur s'affichaient normalement. Migration `20260713101500_fix_project_tags_visibility_teaser_confidential.sql` : ajout de la branche `status = 'confidential' AND (auth.uid() IS NULL OR get_my_role() = 'pending')`, copiée à l'identique de `projects_select_unified`. Les 3 policies de jonction reflètent désormais exactement les 4 branches de visibilité de `projects_select_unified` (public, validated_visitor, teaser pending/anon, accès approuvé) plutôt que d'en couvrir un sous-ensemble.
- `get_advisors` (security) vérifié après les quatre migrations : aucune nouvelle alerte, les warnings existants (buckets/fonctions `security definer` documentées ailleurs) sont inchangés. `tsc --noEmit` propre après le nettoyage client.

## F-24 RGPD — droit à l'effacement, self-service (12/07) — terminée

Feature Must du PRD avec critère d'acceptation explicite, implémentée avant mise en ligne (vendredi). Suite directe de la session précédente : le placeholder "bouton Supprimer mes données" retiré de la politique de confidentialité (fonctionnalité inexistante à l'époque) est maintenant réellement implémenté.

- **Étape 0 (schéma, vérifiée avant tout code)** : introspection de la base live (`list_tables` verbose) — **aucune contrainte FK réelle entre `user_profiles.id` et `auth.users.id`** (lien uniquement applicatif via le trigger `handle_new_user()`), mais **`access_requests.user_id` et `contacts.user_id` portent bien une FK réelle vers `user_profiles.id`** (delete_rule par défaut = RESTRICT). Conséquence directe sur la conception : `user_profiles` ne peut pas être supprimée tant que ces lignes existent → **anonymisation en place, jamais suppression de la ligne**. La table `contacts` porte d'ailleurs déjà en commentaire de colonne le pattern d'anonymisation attendu ("Anonymise -> Utilisateur supprime sur demande RGPD" / "-> null") — signe que cette feature était prévue dès la conception du schéma, jamais câblée côté app.
- **Edge Function `anonymize-rgpd`** (`supabase/functions/anonymize-rgpd/index.ts`) : self-service, jamais admin — cible toujours `auth.getUser()` du JWT appelant, aucun id accepté en paramètre (empêche par construction qu'un utilisateur supprime les données d'un autre). 4 étapes avec service_role : (1) anonymise `user_profiles` (email → `deleted-{uuid}@anonymized.invalid` pour respecter la contrainte UNIQUE, full_name → "Utilisateur supprimé", company/request_message/rejection_reason → null), (2) anonymise `access_requests.message` en conservant status/rejection_reason/project_id (conforme à la politique de confidentialité : "la raison du refus est conservée... mais le message initial peut être supprimé"), (3) anonymise `contacts` (name/email/message), (4) supprime le compte `auth.users` via `admin.deleteUser` — en dernier, pour ne couper l'accès qu'une fois le reste acquis (pas une contrainte technique puisqu'aucune vraie FK ne relie les deux, juste une précaution métier).
- **Bouton "Supprimer mes données"** dans `AccountPage.tsx`, visible pour tous les utilisateurs qui atteignent cette page (pending et validated_visitor — les admins en sont de toute façon redirigés ailleurs, aucune restriction de rôle supplémentaire nécessaire). Modale de confirmation dédiée (action irréversible), pas de suppression en un clic malgré ce que suggérait l'ancien texte retiré de la politique — jugé plus sûr et plus cohérent avec le reste de l'app (mêmes patterns de confirmation que `ProjectDrawer`/`AdminPage`).
- **Bug de course corrigé en testant** : `AccountPage` a son propre `useEffect` qui redirige vers `/auth` dès que `session` devient `null` — le `supabase.auth.signOut()` de `handleDelete()` déclenchait cet effet en concurrence avec le `navigate("/")` explicite qui suit, atterrissage non déterministe (observé : `/auth` au lieu de `/`). Fix : flag `justDeleted` qui désarme l'effet le temps de la redirection volontaire.
- **Testé de bout en bout en conditions réelles, deux fois** (vrais comptes Gmail `+alias`, pas de mock) : inscription via `AccessRequestModal` → demande d'accès avec message → suppression depuis `/account` → vérifié en base : `user_profiles` anonymisé, `access_requests.message` null (status/project_id conservés), `auth.users` introuvable (requête vide). Deuxième passage after-fix : redirection confirmée vers `/lea-martin` (plus jamais `/auth`).

## Fix regression statut : creation Public/Confidentiel silencieusement sauvee en Brouillon (12/07) — terminée

Bug remonte par l'utilisateur en testant en conditions reelles (pas un mock) : creer un nouveau projet, choisir un statut Public ou Confidentiel, cliquer "Enregistrer et publier" -- aucune modale de confirmation ne s'affichait, et le projet finissait enregistre en **Brouillon** malgre le statut visuellement selectionne.

**Cause racine** : `emptyProject()` (`ProjectDrawer.tsx`) initialisait `status: "draft"` -- mais en creation, `statusOptions` exclut deliberement l'option "Brouillon" du `<select>` Statut ("Enregistrer comme brouillon" est un bouton separe, cf. section CRUD projets plus bas). Consequence : le `<select>` React n'a **aucune `<option value="draft">`** a assigner, donc le navigateur affiche visuellement la premiere option du DOM ("Public") alors que l'etat React reel (`draft.status`) reste "draft" tant que l'admin n'a pas explicitement cliqué dans le menu -- un mensonge visuel silencieux. Si l'admin ne touche pas au select (il voit deja "Public" affiche, comportement normal de ne pas y toucher), le clic sur "Enregistrer et publier" lit `draft.status` (l'etat reel, "draft") et non ce qui est affiche a l'ecran.

**Deux bugs lies, corriges ensemble** :
1. `emptyProject()` initialise desormais `status: "public"` (au lieu de `"draft"`) -- coherent avec la premiere `<option>` reellement disponible en creation, l'etat React et l'affichage du select ne divergent plus jamais.
2. La confirmation de changement de statut (F-05) ne se declenchait **jamais** en creation (`{pendingSave && project && (...)}` exigeait un `project` non-null, et `submitWithStatus` ne testait `targetStatus !== project.status` que si `project` existait). Publier un projet tout neuf en Public/Confidentiel le rend pourtant immediatement visible -- meme enjeu qu'un changement de statut en edition. Fix : `submitWithStatus` declenche desormais la confirmation aussi en creation, des que le statut cible n'est pas "draft" (`project ? targetStatus !== project.status : targetStatus !== "draft"`) ; le rendu de la modale ne requiert plus `project` (juste `pendingSave`), avec un titre/message distinct pour la creation ("Confirmer la publication ?" / "Ce projet sera créé avec le statut « X » et immédiatement visible...") vs l'edition (inchangé, "current → cible").

**Testé de bout en bout en conditions reelles** (persona Léa, nouveau projet, upload d'une vraie image test, tous les champs obligatoires remplis, **sans jamais toucher au select Statut** -- exactement le scenario du bug) : la modale "Confirmer la publication ?" s'affiche desormais correctement, et apres confirmation le projet est bien sauvegarde avec `status = 'public'` (verifie en base), plus jamais silencieusement en brouillon. Projet de test supprime apres verification.

**A noter** : le projet "Design Lab Iconographie", cree par l'utilisateur lui-meme en reproduisant le bug initial, a ete laisse tel quel (statut passe a `confidential` lors de l'investigation) -- pas un artefact de cette session, ne pas y toucher sans consigne explicite.

## Flux "demande d'accès confidentiel" (F-12) — implémenté, réel (pas un mock)

`ProjectCard.tsx` a 4 états d'accès pour un projet confidentiel — calculés dans `CataloguePage.tsx` (`resolveAccess`) à partir du rôle global (`get_my_role()`) **et** des lignes `access_requests` propres au visiteur connecté (`src/data/accessRequests.ts`) :
- **none** → carte cliquable, ouvre `AccessRequestModal`.
- **pending** → carte inerte, `Alert` type info.
- **refused** → carte inerte, `Alert` type warning + lien vers la section contact du profil.
- **granted** → badge "Confidentiel · Accès validé", carte = lien direct vers la fiche complète.

**Important** : une ligne `access_requests.status = 'approved'` débloque **un seul projet précis** pour ce visiteur, sans changer son rôle global (`validated_visitor` reste le seul rôle qui débloque *tous* les confidentiels — vérifié en base : approuver une demande ne touche jamais `user_profiles.role`, aucun trigger ne le fait non plus). Ce mécanisme est reconnu à deux endroits qu'il faut garder synchronisés si on retouche les RLS : la policy `projects_select_unified` (branche `exists (select 1 from access_requests ...)`) et `getProjectById` (`src/data/projects.ts`, fiche détail complète).

**⚠️ Piège RLS récursion croisée** : `projects_select_unified` interroge `access_requests`, et la policy d'insert sur `access_requests` interroge `projects` (pour vérifier `sensitivity_level`) — deux policies sur deux tables qui se référencent mutuellement en sous-requête directe déclenchent `42P17 infinite recursion detected in policy`. Fix : passer par des fonctions `SECURITY DEFINER` (`project_is_sensible()`, `has_other_approved_access_request()`) qui bypassent RLS, exactement comme `get_my_role()` le fait déjà pour `user_profiles`. Ne jamais remettre une sous-requête brute inter-tables dans une policy ici sans repasser par ce pattern.

Back-office admin (`AdminPage.tsx`, onglet Demandes + dashboard) branché sur les vraies fonctions `src/data/accessRequests.ts` (`getAllAccessRequests`/`approveAccessRequest`/`rejectAccessRequest`) — le mock `seedRequests`/`src/data/requests.ts` a été supprimé, ne plus le chercher.

Les projets `tres_sensible` passent toujours en revue.

`AccessRequestModal.tsx` fait un vrai travail, plus un mock :

**Trois bugs trouvés et corrigés le 11/07 en testant le flux F-12 en conditions réelles** (comptes réels, pas seulement les personas) :

1. **Demandes qui restaient affichées "Demander l'accès" après envoi réussi** — `CataloguePage.tsx` ne rechargeait `myRequests` qu'au montage ; rien ne le redéclenchait après un insert réussi dans `AccessRequestModal`. Fix : `AccessRequestModal` accepte un prop `onSuccess`, appelé juste après l'insertion, que `CataloguePage` branche sur un `refreshMyRequests` qu'elle passe en `useCallback`.
2. **`refreshMyRequests` court-circuitait sur un `session` périmé** — la version initiale du fix ci-dessus gardait `if (!session) return` dans `refreshMyRequests`, avec `session` capturé en closure React. Pour une **inscription anonyme** (visiteur qui crée son compte dans la modale), cette closure reste figée sur `session = null` — capturée au moment où la modale s'est ouverte, anonyme — même après l'inscription réussie, donc `onSuccess()` appelait une version périmée qui vidait `myRequests` au lieu de le recharger. `getMyAccessRequests()` (`src/data/accessRequests.ts`) n'a en réalité besoin d'aucun `session` : il interroge le client Supabase global, toujours à jour côté RLS. **Piège à retenir** : ne jamais gater une fonction passée en callback à un enfant sur un état React capturé en closure si cet enfant peut l'appeler après un changement d'auth asynchrone (signUp) survenu **pendant** sa propre exécution — la closure ne se "rafraîchit" pas au milieu d'un `async function` déjà lancé.
3. **Connexion via `/auth` invisible pour tout non-admin** — `AuthPage.tsx` redirigeait systématiquement vers `/admin` après connexion réussie ; `AdminPage.tsx` (garde `role !== "admin" → navigate("/")`) renvoyait alors silencieusement vers l'accueil quiconque n'est pas admin, donnant l'impression que la connexion avait échoué (aucune erreur, juste retour à la case départ). Fix : `AuthPage` récupère le rôle réel après connexion et route en conséquence (`admin` → `/admin`, sinon → le catalogue, où l'état des demandes est visible immédiatement).
4. **Demandes en double sur un même projet** — la modale listait *tous* les projets confidentiels comme cases à cocher, y compris ceux déjà demandés par le visiteur connecté (seule la carte `ProjectCard` elle-même était gatée). Un visiteur ouvrant la modale depuis un *autre* projet pouvait donc cocher un projet déjà en `pending`, créant une 2e ligne `access_requests` pour la même paire (visiteur, projet). Fix double : `CataloguePage` passe `excludeProjectIds` (dérivé de `myRequests`) à la modale qui filtre son checklist, **et** un index unique partiel en base (`access_requests_one_pending_per_user_project`, migration `20260711032500`) empêche deux lignes `pending` simultanées pour la même paire — une nouvelle demande reste possible une fois la précédente résolue (approuvée/refusée).

## Webhooks Supabase → Resend (PRD 7.3) — 4 webhooks, implémentés le 11/07

Infrastructure **Database Webhooks** construite entièrement en SQL (pas de configuration dashboard) : extension `pg_net` + une fonction trigger générique `public.dispatch_webhook()` (`supabase/migrations/20260711020351_...`) qui relaie tout INSERT/UPDATE vers une Edge Function, avec le payload standard `{type, table, record, old_record}`. Le nom de l'Edge Function cible est passé en argument du trigger (`TG_ARGV[0]`), donc **un seul trigger générique pour les 4 webhooks** plutôt que 4 fonctions trigger dupliquées.

Authentification du trigger → Edge Function : clé publique (`sb_publishable_...`) en dur dans la fonction Postgres — safe, c'est la même clé déjà exposée côté client, aucun secret à protéger ici. Chaque Edge Function (`verify_jwt: true`) interroge ensuite la base en `service_role` (`SUPABASE_SERVICE_ROLE_KEY`, injecté automatiquement par Supabase pour toute Edge Function — pas besoin de le configurer) pour résoudre les emails/noms/titres nécessaires au template.

**Les 4 webhooks** (`supabase/functions/webhook-*/index.ts`, triggers dans `20260711020735_...`) :
1. `trg_webhook_welcome_email` — INSERT `user_profiles` → email "Bienvenue" au nouvel inscrit.
2. `trg_webhook_access_request_created` — INSERT `access_requests` → email visiteur ("Demande en cours") + email à chaque admin ("Nouvelle demande").
3. `trg_webhook_access_request_updated` — UPDATE `access_requests` **seulement** sur la transition `pending → approved`/`rejected` (`WHEN` clause du trigger) → email visiteur "Accès accordé" ou "Refusé" + raison si présente.
4. `trg_webhook_contact_created` — INSERT `contacts` → email de confirmation au visiteur + notification à chaque admin.

**⚠️ Piège Resend sans domaine vérifié** : Resend refuse (`403 restricted_api_key`... non, `403 validation_error`) l'envoi vers une adresse `+alias` tant qu'aucun domaine n'est vérifié sur le compte — il fait une comparaison **stricte de chaîne** avec l'adresse exacte du compte Resend, sans awareness du `+tag` Gmail (testé en direct, pas supposé). Fix appliqué dans chaque Edge Function : `normalizeGmail()` strip le `+tag` avant l'appel Resend, **uniquement** pour `gmail.com`/`googlemail.com` (comportement correct même en prod, puisque c'est réellement la même boîte pour ces domaines — ne pas étendre à d'autres domaines où `+` n'est pas forcément un alias).

**⚠️ Clé API Resend "Sending access" uniquement** : la clé configurée (secret Edge Functions `RESEND_API_KEY`) a été créée avec la permission "Sending access", pas "Full access" — elle peut `POST /emails` mais `GET /emails/{id}` renvoie `401 restricted_api_key`. Conséquence : impossible de vérifier via l'API si un email a réellement été *délivré* (vs juste accepté par Resend), seulement que Resend l'a accepté pour envoi. À changer pour "Full access" si un vrai diagnostic de délivrabilité est nécessaire un jour.

**Domaine non vérifié = déclarativement du mode test** : tous les emails partent actuellement de `onboarding@resend.dev` (domaine partagé Resend, faible réputation, forte proba spam côté Gmail). Vérifier un domaine réel chez Resend est un prérequis avant toute mise en prod — pas juste pour lever la restriction `+alias` des tests.

Testé de bout en bout (SQL réel déclenchant les triggers, pas d'appel direct aux Edge Functions) pour les 6 emails + la variante refusé du webhook 3 — tous acceptés par Resend (`200`, id de message), 0 échec silencieux détecté dans les logs `net._http_response`/Edge Functions.
